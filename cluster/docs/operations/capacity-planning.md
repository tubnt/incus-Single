## 2. 节点数 vs VM 数 vs IP 数计算

### 2.1 VM 数量上限

**约束因素（取最小值）：**

| 约束 | 计算 | 上限 |
|------|------|------|
| IP 数量 | 24 个可用 IP | **24 台 VM** |
| CPU | 80 核 × 80% 可售 = 64 vCPU | 取决于 VM 规格 |
| 内存 | (320 - 40) GiB = 280 GiB 可售 | 取决于 VM 规格 |
| 存储 | 5.3 TiB 可用（见下文） | 取决于 VM 磁盘大小 |
| Incus 项目配额 | 50 实例 / 100 CPU / 200 GiB RAM / 5 TiB disk | 项目级硬限制 |

### 2.2 典型 VM 规格对应数量

| VM 规格 | vCPU | 内存 | 磁盘 | 最大 VM 数 | 瓶颈 |
|---------|------|------|------|-----------|------|
| 入门级 | 1 | 1 GiB | 20 GiB | **24** | IP |
| 标准级 | 2 | 4 GiB | 50 GiB | **24** | IP |
| 高配级 | 4 | 8 GiB | 100 GiB | **24** | IP |
| 旗舰级 | 8 | 16 GiB | 200 GiB | 17 | 内存（280/16） |

> 当前子网下 **IP 是首要瓶颈**，最多 24 台 VM。

### 2.3 HA 容量预留

为保证单节点故障时其余节点能承载所有 VM：

```
有效容量 = 总容量 × (N-1)/N
5 节点集群：有效容量 = 总量 × 80%
```

| 资源 | 总量 | HA 预留后可售 |
|------|------|-------------|
| CPU | 64 vCPU | 51 vCPU |
| 内存 | 280 GiB | 224 GiB |
| 存储 | 5.3 TiB | 5.3 TiB（Ceph 自动处理） |

## 3. Ceph 存储容量规划

### 3.1 容量计算（3 副本）

```
裸容量 = 5 节点 × 4 OSD × 1 TB = 20 TB
有效容量 = 裸容量 ÷ 副本数 × 安全系数
         = 20 TB ÷ 3 × 0.80
         = 5.33 TB（可用）
```

**安全系数 0.80 的原因：**
- Ceph 在 85% 使用率时进入 nearfull 状态
- 需要预留空间用于 recovery/rebalance
- 建议不超过 75% 实际使用率

### 3.2 VM 磁盘容量对照表

| VM 磁盘规格 | 最大 VM 数 | 总使用量 | Ceph 使用率 |
|-------------|-----------|----------|------------|
| 20 GiB × 24 | 24 | 480 GiB | 9% |
| 50 GiB × 24 | 24 | 1.2 TiB | 22% |
| 100 GiB × 24 | 24 | 2.4 TiB | 45% |
| 200 GiB × 24 | 24 | 4.8 TiB | 90% ⚠️ 超过 nearfull(85%)，Ceph 将限制写入 |

### 3.3 容量增长预测

监控以下 Prometheus 指标评估趋势：
- `ceph_cluster_total_used_bytes` — 已使用总量
- `ceph_cluster_total_bytes` — 总裸容量
- `ceph_osd_stat_bytes_used` — 单 OSD 使用量

```
每月增长率 = (本月使用量 - 上月使用量) / 上月使用量
预计达到 75% 的月份 = (75% × 总量 - 当前使用量) / 月增量
```

## 4. 带宽规划

### 4.1 网络拓扑带宽

| 网络 | 物理带宽 | MTU | 用途 |
|------|----------|-----|------|
| eno1（untagged / br-pub） | 10 Gbps（共享） | 1500 | VM 公网流量 |
| eno1 VLAN 10 | 10 Gbps（共享） | 1500 | 管理网 |
| eno1 VLAN 20 | 10 Gbps（共享） | 9000 | Ceph Public |
| eno2 | 10 Gbps（专用） | 9000 | Ceph Cluster 复制 |

### 4.2 VM 带宽分配

当前默认：每 VM 100 Mbps 双向。

| VM 数量 | 峰值总带宽需求 | 单节点承载 | 是否超出 |
|---------|--------------|-----------|---------|
| 10 | 1 Gbps | 2 Gbps（5 台分摊） | 否 |
| 20 | 2 Gbps | 4 Gbps（5 台分摊） | 否 |
| 24 | 2.4 Gbps | 4.8 Gbps（5 台分摊） | 否 |

> eno1 10Gbps 需同时承载 VM 流量 + VLAN 10/20，但 Ceph 复制走 eno2，因此 VM 带宽上限约 6-7 Gbps/节点。24 台 VM 的峰值带宽在安全范围内。

### 4.3 Ceph 带宽需求

| 场景 | 带宽需求 | 网络 |
|------|---------|------|
| 正常 VM IO | < 1 Gbps | Ceph Public (VLAN 20) |
| OSD 故障 recovery | 3-5 Gbps | Ceph Cluster (eno2) |
| 节点故障全量恢复 | 最高 10 Gbps | Ceph Cluster (eno2) |

Recovery 流量走专用 eno2，不影响 VM 公网和管理网流量。

## 5. 扩容触发条件

### 5.1 告警阈值与扩容决策

| 资源 | Warning（关注） | Critical（行动） | 扩容方式 |
|------|----------------|-----------------|----------|
| Ceph 存储 | > 75% | > 85% | 加 OSD 磁盘或加节点 |
| 单 OSD | > 80% | > 90% | reweight 或加 OSD |
| IP 池 | 剩余 < 20%（< 5 个） | 剩余 < 10%（< 3 个） | 采购新 IP 段 |
| CPU | > 80% 持续 10min | > 95% 持续 5min | 迁移 VM 或加节点 |
| 内存 | > 85% | > 95% | 迁移 VM 或加节点 |
| 节点数 | 单节点 VM > 8 | 单节点 VM > 12 | 加节点 |

### 5.2 存储扩容流程

#### 方案 A：现有节点加 OSD 磁盘

```bash
# 1. 安装新 SSD 到目标节点
# 2. 确认磁盘可见
lsblk

# 3. 添加为 OSD
cephadm shell -- ceph orch daemon add osd <host>:/dev/<new-disk>

# 4. 等待数据 rebalance
ceph -w

# 5. 验证新 OSD 正常
ceph osd tree
```

**容量变化**：每增加 1 块 1TB SSD = 增加 ~266 GiB 可用空间（÷3 副本 × 0.80）

#### 方案 B：添加新节点

```bash
# 1. 硬件准备：网络配置、系统安装
# 2. 使用 join 脚本加入集群
bash scripts/join-node.sh <新节点名> <新节点IP>

# 脚本自动完成：
# - 网络配置（netplan）
# - 加入 Incus 集群
# - 添加 Ceph OSD
# - 配置防火墙规则
# - 部署监控 agent

# 3. 验证
incus cluster list
ceph osd tree
```

**容量变化**：每增加 1 台标准节点 = 增加 ~1.07 TiB 可用空间 + 16 核 CPU + 64 GiB 内存

### 5.3 IP 扩容流程

1. 向 ISP 申请新的公网 IP 段
2. 交换机配置新子网路由
3. 各节点 nftables 规则更新（如需要）
4. Paymenter 数据库添加新 IP 池
5. 测试新 IP 段网络连通性

### 5.4 带宽扩容

| 方案 | 适用场景 | 操作 |
|------|---------|------|
| 提升单 VM 带宽上限 | 高带宽产品需求 | 修改 Incus 带宽限制 |
| 增加上行带宽 | 总带宽不足 | ISP 升级或多线接入 |
| Link Aggregation | 单链路瓶颈 | 网卡 bonding（需交换机支持） |

---

## 6. 扩容规划建议

### 6.1 短期（当前 /27 子网内）

- 最大 24 台 VM
- 存储上限约 5.3 TiB（50 GiB/VM 时足够 24 台）
- CPU/内存在 HA 预留后支持 24 台标准配置 VM

### 6.2 中期扩容路径

| 阶段 | 触发条件 | 扩容内容 | 预计成本 |
|------|---------|---------|---------|
| 第一步 | IP 使用 > 80% | 采购新 /27 或 /28 子网 | IP 费用 |
| 第二步 | 存储 > 75% | 每节点各加 2 块 SSD | 10 × SSD |
| 第三步 | CPU/内存瓶颈 | 增加第 6、7 节点 | 服务器 × 2 |

### 6.3 容量监控仪表板

建议在 Grafana 中配置以下面板：

| 面板 | 数据源 | 关键指标 |
|------|--------|---------|
| Ceph 容量趋势 | Prometheus | `ceph_cluster_total_used_bytes / ceph_cluster_total_bytes` |
| 节点 CPU/内存 | Prometheus | `node_cpu_seconds_total`, `node_memory_MemAvailable_bytes` |
| IP 池余量 | Paymenter DB | 可用 IP 数 / 总 IP 数 |
| VM 数量趋势 | Incus metrics | 实例计数按状态分组 |
| 带宽使用 | Prometheus | `node_network_transmit_bytes_total{device="eno1"}` |
