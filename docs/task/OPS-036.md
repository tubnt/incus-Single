## 实际改动

### A. Command Palette 升级 ✓
`shared/components/command-palette/command-palette.tsx`：
- `pages` 单 group → `pageGroups` 多 group（按 sidebar section 分块，每个 section 独立 CommandGroup heading）
- 引入 `Fragment` 包裹多 group 渲染 + `CommandSeparator`
- 用户视角是第一组（仪表盘/我的云主机/SSH 密钥/账单/工单/API/设置）
- 管理员视角后接 5 个分组（监控/资源/基础设施/计费/用户运维）

### B. Tickets detail page —— 跳过
**理由**：实测当前 expand row 模式是 Linear 风（点击行展开内联详情），用户体验比 GitHub detail page 还好（不离开列表上下文）。改为 detail page 会增加跳转成本，**投资回报为负**。

### C. Sidebar 折叠非活动分组 —— 跳过
**理由**：审查 `app-sidebar.tsx` 现有逻辑：进入路由时自动展开活动分组 + 保留用户手动展开（B 模式），已合理。无需改动。

## 重新评估的 IA 现状

OPS-035 完成后，admin sidebar 实际是 **20 项分 5 组**（不是估错的 30+）。分组语义清晰：
1. 监控 (3) — monitor / observability / ha
2. 资源 (3) — vms / create / storage
3. 基础设施 (7) — clusters / nodes / nodeOps / ipPools / ipRegistry / firewall / floatingIPs
4. 计费 (4) — products / osTemplates / orders / invoices
5. 用户运维 (3) — users / adminTickets / auditLogs

**用户当时反馈"复杂"主要源于 OPS-035 之前的字号偏大**，IA 本身已合理。再做 IA 重构（admin/* 整组归 Settings 二级 tabs / 顶级 sidebar 收 7 项 / detail page 推广）属于过度工程，建议**保留现状**。
