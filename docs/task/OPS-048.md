# OPS-048 Incus server cert 重签含 wireguard SAN + 重启 W2 fail-fast

- **status**: backlog
- **priority**: P2
- **owner**: TBD
- **createdAt**: 2026-05-10
- **scope**: 修复部署期发现的 cert SAN 与 wireguard IP 不符问题，让 PLAN-051 W2 production fail-fast 能在 vmc.5ok.co 重新打开。

## 现状（2026-05-10 部署后实测）

PLAN-051 §2-A 引入的 `requireProductionSecurity()` 会校验
`clusters[*].CAFile` 必填。在生产 vmc.5ok.co 部署时打开 `CLUSTER_CA_FILE=
/etc/incus-admin/certs/ca.crt`，立刻撞 mTLS 严格校验失败：

```
"failed to connect cluster","name":"cn-sz-01",
"error":"...x509: certificate is valid for 127.0.0.1, ::1, not 10.0.20.1"
```

`/etc/incus-admin/certs/ca.crt` 是 Incus 默认 self-signed CA + server cert。
默认 cert SAN 只有 `127.0.0.1, ::1`，但 incus-admin 通过 wireguard 隧道连
`https://10.0.20.1:8443`，标准 Go TLS verify SAN 不匹配 → 直接断。

之前历史部署没配 `CLUSTER_CA_FILE` → 走 `InsecureSkipVerify=true` + SPKI
TOFU pin（`clusters.tls_fingerprint` DB 列）→ 不校验 SAN → 能连。

## 临时绕过（已上线）

`/etc/incus-admin/incus-admin.env`：
- `INCUS_ADMIN_ENV=staging`（绕开 W2 fail-fast）
- `# CLUSTER_CA_FILE=/etc/incus-admin/certs/ca.crt`（注释掉 → 退回 SPKI TOFU）

W1 + W3 仍生效（SSH_KNOWN_HOSTS_FILE / PASSWORD_ENCRYPTION_KEY 强约束）。

## 真修复（本任务）

**Option A（推荐）**：重签 Incus server cert 含 wireguard SAN
1. 在 cluster 端（10.0.20.1）跑：
   ```bash
   incus config trust list
   # 找到 incus 服务自身的 keypair（通常在 /var/snap/incus/common/incus/server.crt + server.key）
   # 用 openssl 重签含 SAN：
   openssl req -new -key server.key \
     -subj '/CN=incus-cluster-cn-sz-01' \
     -addext 'subjectAltName=DNS:incus.local,IP:10.0.20.1,IP:127.0.0.1,IP:::1' \
     -out server.csr
   openssl x509 -req -in server.csr -signkey server.key \
     -copy_extensions copyall -days 3650 -out server.crt
   incus admin shutdown / restart
   ```
2. 把新 server.crt 同步到 incus-admin 主控的 `/etc/incus-admin/certs/ca.crt`
3. `INCUS_ADMIN_ENV=production` + 取消 `CLUSTER_CA_FILE` 注释
4. `systemctl restart incus-admin` 验证 W2 fail-fast pass + cluster connect ok

**Option B**：把 cert SAN 改用 DNS 名 + hosts 别名
- `/etc/hosts` 加 `10.0.20.1 incus-cluster.l.5ok.co`
- 重签 cert SAN=`incus-cluster.l.5ok.co`
- `CLUSTER_API_URL=https://incus-cluster.l.5ok.co:8443`

**Option C（低优先 RFC）**：代码层面加 `CLUSTER_CERT_SKIP_SAN_VERIFY=true`
显式 opt-in，让 TOFU pin 完全替代 SAN 校验（已有 SPKI pin store 实现）。

## 风险

- 重签 cert 期间 cluster 短暂不可达（建议低峰期；预计 < 30s）
- 旧 client 连接需要丢弃（旧 SPKI pin 失效）；incus-admin 主控的
  `clusters.tls_fingerprint` 行需要清空让其重新 TOFU 学习
- 节点之间集群通信可能也用同一证书；要核对 cluster 节点 trust list

## 验收

```bash
# .env 恢复 production 形态
INCUS_ADMIN_ENV=production
CLUSTER_CA_FILE=/etc/incus-admin/certs/ca.crt

systemctl restart incus-admin
# 期望：service active + cluster connected + 0 error 日志
journalctl -u incus-admin --since "1m ago" | grep -E '(cluster connected|certificate)'
```
