## 描述

用户自服务 L4 安全组，封装 Incus network ACL。每组 DB → 一个 `fwg-<slug>` ACL（落在 `default` project，customers 继承 networks）；VM NIC 的 `security.acls` 逗号列表决定应用哪些组。

## 交付

- migration 011 三表（firewall_groups / firewall_rules / vm_firewall_bindings）+ 3 组 seed（web / ssh-only / database-lan）
- repository（groups + rules + bindings 三段）+ service（EnsureACL / DeleteACL / AttachACLToVM / DetachACLFromVM，均幂等）
- admin handler CRUD + portal bind/unbind（owner 校验）
- 前端 admin /admin/firewall 页面 + sidebar 入口 + en/zh i18n
- 单测 6 组 23 case（slug→ACL name / rule 转换 / NIC ACL 列表合并 / NIC 设备选择）
- 生产 E2E 6 合约测试全 PASS，Incus ACL 真建/改/删 + audit 3 行留档

## 笔记

- ACL 项目选 `default`：customers 项目 NETWORKS=NO 继承默认项目，per-project ACL 会 collapse。实测 `--project customers` 创建的 ACL 实际落在 default
- rule `state` 必填（"enabled"），Incus 6.23 强校验
- 写入策略：DB 先提交，Incus sync 失败返 202 + `sync_err` 让 admin 看到且可重试（编辑规则自动重新 EnsureACL）
- 删除顺序：先 Incus ACL（吞 404 幂等）再 DB；万一 Incus 失败 DB 仍删掉，留个 stale ACL 给 admin 手清
- NIC device 选择：`eth0` 优先，fallback 首个 `type=nic` 的设备
- 只做 L4（protocol + destination_port + source_cidr），L7 WAF 不在范围内
- 暂未做 ingress/egress 双向（只 ingress），需求浮现再扩
