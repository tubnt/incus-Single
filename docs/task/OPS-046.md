## 背景

OPS-045 仅 seed 了 Windows OS 模板占位（enabled=false），用户实际选择 Windows 创建云主机走不通 —— UI 流程和后端 vm_create 都按 Linux 路径处理，且 cluster 公网 VLAN 376 桥配置在 OPS-002 落地时有半截缺失（pub.376 子接口建了但未绑入 br-pub），所有 VM（不只 Windows）从 br-pub 出口都收不到上游网关 ARP 回包。

本任务把 Windows 模板从占位推到 **真能用 ai@5ok.co 通过 /launch 创建出来 → IP 自动配 → 公网 RDP 直连** 的完整闭环，同时修复 OPS-002 遗漏的 br-pub VLAN 绑桥问题（顺带让所有 Linux VM 公网也通）。

## 改动范围

### 1. Windows 镜像 build pipeline（社区方案）

不再用任何 prebuilt 第三方 Windows 镜像（cloudbase 2017 prebuilt 实测 sysprep state 0x7 死循环）。改用 GitHub `antifob/incus-windows` 一站式构建：

```
sh /opt/incus-windows/build.sh -i isos/2022.iso 2022
sh /opt/incus-windows/tools/import.sh output/win2022/
incus image alias rename win2022 windows-server-2022
incus image copy local:windows-server-2022 local: --target-project customers --alias windows-server-2022
```

镜像特点：
- UEFI 原生（不用 CSM）
- Image properties `requirements.cdrom_agent=true`，需要 instance 显式挂 `agent:config` disk
- 内置 incus-agent，可走 vsock 通道执行 PowerShell（无 cloudbase-init）
- 默认 admin/changeme 用户已建好，RDP 服务装好但默认未启

工具 `pack.sh` 改了 2 行（生产 cluster 的 default profile 没 root device，原脚本 `-d root,size=30GiB` 会报"override fails"）：
- `incus init ... --empty --vm -s ceph-pool`（指定 storage pool）
- `incus config device set "${name}" root size=30GiB`（在 -s 自动建的 root 上 set，不是 device override）

### 2. Backend OS-aware（`internal/service/jobs/vm_create.go`）

`isWindowsAlias()` 检测 OS 类型；Windows 路径相对 Linux 多做以下事：

- 预生成 MAC（OUI `10:66:6a:`，与 Incus 自带 Linux VM 一致；早期试过 QEMU 标准 `52:54:00:` 上游交换机不识别）；MAC 绑到 nic device + 写进 cloud-init network-config
- network-config 用 v1 + `mac_address` 匹配（cloudbase-init NetworkConfigPlugin 不支持 v2 接口名匹配，但我们 antifob 镜像没 cloudbase-init，这条主要给未来其他镜像用）
- Instance device 必加 `cloud-init:config` (cidata) + `agent:config` (incus-agent ISO，无此 image start 直接报 "requires an agent:config disk be added")
- alias→fingerprint 预解析（receive 受限 cert 创建实例时 alias 不可见，必须先 GET 拿 fingerprint）
- finalize 步骤新增 `applyWindowsCloudInit()`：通过 incus-agent vsock 通道跑 PowerShell 配静态 IP/网关/DNS/RDP/Administrator 密码（antifob 镜像无 cloudbase-init，靠这个兜底）；先 `waitWindowsAgent` 轮询 `state.processes>0` 最长 5min + 60s buffer 等 NetSetup 就绪
- 跳 `applyUserDefaultFirewallGroups`：Windows VM 用户的默认 fwg 通常是 ssh-only，会挡 3389；让用户事后到 /firewall 按需挂

### 3. Backend HTTP timeout（`internal/cluster/manager.go`）

短 client.Timeout 10s → **30s**。10s 在跨 wg 隧道的 TLS 握手 + Incus 内部 DB 写偶尔被误杀 → 经 vm.create job submit_instance 步直接失败 + 退款（实测线上观察到 3 次 "context deadline exceeded"）。

### 4. Reinstall 路径同步修

`internal/handler/portal/reinstall_resolve.go::defaultUserForSource` 加 `windows` 分支返回 `Administrator`。`internal/service/vm.go::Reinstall` 用同样的 alias 含 `/` 启发式区分 simplestreams vs 本地 Incus image。

### 5. UI（前端 OS picker）

`web/src/features/vms/os-image-picker.tsx` 加 Windows 分组 + `familyFor()` 单段 alias 兜底；`default-user.ts` 加 `windows` → `Administrator` 映射。

### 6. DB migration 023

`db/migrations/023_windows_os_template.sql` seed `windows-server-2022` / `windows-11` 两条占位 + Windows 11 enabled=false。生产手工把 `windows-server-2022` enabled=true 后即可见。

### 7. **关键修复：cluster 公网 VLAN 376 绑桥**（影响所有 VM 不只 Windows）

OPS-002 遗漏：`/etc/netplan/60-vlan376.yaml` 建了 `pub.376` VLAN 子接口，但**没人绑进 br-pub bridge**。所以所有 VM 出 br-pub 后流量是 untagged（PVID=1），上游交换机给 VM /26 段是 VLAN 376 tagged → ARP 回包被 br-pub vlan_filtering 丢弃。结果 Linux + Windows 所有 VM 全部 ARP gateway 收不到回包，公网 100% 不通（实测 vm-4609c0 Ubuntu 也不通）。

**修复**：5 节点全跑：
```bash
sudo ip link set enp130s0f0np0 nomaster   # 把误绑的 raw 物理口踢出（仅 node1 需要）
sudo ip link set pub.376 master br-pub    # VLAN 376 子接口绑桥
sudo incus network set br-pub --target <node-name> bridge.external_interfaces=pub.376  # 持久化
```

**部署脚本同步改**（避免下次加节点重蹈覆辙）：
- `cluster-env.sh` 加 `VLAN_PUB=376` / `VM_PUBLIC_NETWORK="202.151.179.0/26"` / `BRIDGE_EXTERNAL_IFACE="pub.${VLAN_PUB}"`
- `join-node.sh` netplan 模板加 `pub.${VLAN_PUB}` 子接口；br-pub interfaces 从 raw `${NIC_PRIMARY}` 改 `pub.${VLAN_PUB}`；Incus join 后自动 `incus network set br-pub --target <node> bridge.external_interfaces=pub.${VLAN_PUB}`；最终验证步骤加"VLAN 子接口存在 + 已绑桥"两条
- `apply-network.sh` 自检加 "检查 6"：`pub.${VLAN_PUB}` 必须是 br-pub 的 slave，否则触发 5 分钟自动回滚

## 笔记

发现 OPS-002 遗漏过程：从 ai@5ok.co 创建 Windows VM 公网不通开始排查，验证 Linux VM 同样不通排除"Windows 特定问题"假设；tcpdump enp130s0f0np0 物理口看到上游 ARP 全是 vlan 376 tagged；查 br-pub vlan_filtering=1 PVID=1 无 vlan 376 → 推到 pub.376 子接口未绑桥。
