# OPS-022 vms.password DB 字段加密（AES-256-GCM）

- **status**: completed
- **completedAt**: 2026-04-30（生产 22 条历史 vms.password 已加密，DB 查询 100% v1: 前缀）
- **priority**: P2
- **owner**: claude
- **createdAt**: 2026-04-30

## Description

`vms.password` 当前明文存 DB，admin 直接 `psql` 查询或 `pg_dump` 备份能看到所有用户密码。前端 `SecretReveal` 文案"密码仅显示一次"对用户是误导。

## 范围

- `internal/auth/password_crypto.go`：AES-256-GCM 加解密 helper（`EncryptPassword` / `DecryptPassword`）。Key 从 `INCUS_PASSWORD_ENCRYPTION_KEY` env 读取，base64 编码 32 字节。版本前缀 `v1:` 给 rotation 留口。
- `VMRepo` 写路径（`Create` / `UpdatePassword` / `UpdateAfterProvision`）加密后入 DB；读路径解密；解密失败 → password 字段返 nil + warn。
- migration 018：walk `vms WHERE password NOT NULL AND password NOT LIKE 'v1:%'` → 在 Go 内 encrypt → UPDATE。Idempotent（已加密的跳过）。直接在应用启动时跑一次（不用 SQL 表达式因 key 在 env 里）。
- main.go：启动调 `auth.SetPasswordEncryptionKey(cfg.Auth.PasswordEncryptionKey)`，再跑历史数据 migration。
- config 加 `PASSWORD_ENCRYPTION_KEY` 字段（在 cfg.Auth 下）。
- 加密未启用（env 空）→ passthrough：所有 helper 不动密文，向后兼容老部署。

## 不在范围

- key rotation（OPS-023 立项）：v1 设计就是 single-key + 版本前缀，rotation 升 v2 时新增 key 字典 + 双向 fallback
- per-row salt
- 客户端侧加密（off-DB key store）

## ActiveForm

vms.password DB 字段加密（AES-256-GCM）

## Acceptance criteria

- 配 `INCUS_PASSWORD_ENCRYPTION_KEY` 后重启：`SELECT password FROM vms` 返回 `v1:<base64>` 形式，`pg_dump` 看不见明文
- 前端 SecretReveal 仍能正常显示新密码（解密透明发生在 repo 层）
- 不配 key 时服务正常启动 + 旧明文行可读 + 新写入还是明文（passthrough）
- 配了 key 但加密 fail → 后端 log warn，不阻塞业务（password 字段返 nil）
- `go vet` clean / `go test` 全绿；新增 password_crypto_test 覆盖正常 / 错误 key / 旧明文 passthrough
