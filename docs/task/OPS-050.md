# OPS-050 jobs runtime 容量可配 + 队列深度可观测

- **status**: completed
- **priority**: P2
- **owner**: claude
- **createdAt**: 2026-05-11
- **completedAt**: 2026-05-11
- **scope**: 异步 provisioning runtime 的 `PoolSize=4` / `QueueSize=64` 当前
  在 `cmd/server/main.go:433` 与 `runtime.go:88` 写死，促销 / 批量场景下
  并发 64+ 用户下单会让 `Enqueue` 阻塞 HTTP 请求（PayWithBalance 已 commit、
  余额已扣的状态下挂死）。本任务把两者拆到 env，加运行期 queue 深度日志，
  并在任务文档里附孤儿订单恢复 Runbook。

## 方案（最小集，单一 PR）

### 1. 新增 `JobsConfig`

```go
// internal/config/config.go
type JobsConfig struct {
    PoolSize  int // env JOBS_POOL_SIZE，默认 4
    QueueSize int // env JOBS_QUEUE_SIZE，默认 64
}
```

`Config.Jobs` 字段、`Load()` 里 `parseIntOr` 读两条 env。

### 2. main.go 接线

```go
jobsRuntime = jobs.NewRuntime(jobs.Deps{
    ...
    PoolSize:  cfg.Jobs.PoolSize,
    QueueSize: cfg.Jobs.QueueSize,
})
slog.Info("provisioning jobs runtime started",
    "pool_size", cfg.Jobs.PoolSize,
    "queue_size", cfg.Jobs.QueueSize)
```

### 3. runtime.go 三项改动

- `Deps` 加 `QueueSize int`（0 取默认 64）。
- `NewRuntime` 用 `deps.QueueSize` 建 buffered channel。
- `Start` 起一个 metrics ticker goroutine：每 30s 仅当 `len(r.queue) > 0` 时
  打一行 `provisioning queue depth depth=N pool_size=M`。空载不刷屏。
- 加 `QueueDepth()` 方法返 `len(r.queue)`（给未来 metrics endpoint 留口）。

### 4. .env.example

加注释 + 默认值：

```
# 异步 provisioning runtime（OPS-050）
# 生产建议 8 / 256；默认 4 / 64 对应单机小流量
JOBS_POOL_SIZE=4
JOBS_QUEUE_SIZE=64
```

### 5. Runbook：孤儿订单恢复

详见下方 §Runbook。手动 SQL，不写代码。

## Runbook：孤儿订单恢复

**触发条件**：`orders.status='paid' AND` 没有对应 `vms.order_id` 行 `AND`
没有对应 `provisioning_jobs.order_id` 行。窗口极窄（PayWithBalance commit
后 / Enqueue 前进程被 SIGKILL，约 ms 级），但生产仍可能撞上。

**诊断 SQL**（只读）：

```sql
SELECT o.id, o.user_id, o.amount, o.status, o.created_at
FROM orders o
LEFT JOIN vms v ON v.order_id = o.id
LEFT JOIN provisioning_jobs j ON j.order_id = o.id
WHERE o.status = 'paid'
  AND v.id IS NULL
  AND j.id IS NULL
  AND o.created_at < NOW() - INTERVAL '10 minutes'
ORDER BY o.created_at DESC;
```

**修复 SQL**（事务包裹，先 dry-run 再 commit）：

```sql
BEGIN;
-- 1) 退余额
UPDATE users SET balance = balance + (SELECT amount FROM orders WHERE id = $ORDER_ID)
WHERE id = (SELECT user_id FROM orders WHERE id = $ORDER_ID);
-- 2) 记 transaction
INSERT INTO transactions (user_id, amount, type, description)
SELECT user_id, amount, 'refund', '订单 #' || id || ' 孤儿恢复 OPS-050'
FROM orders WHERE id = $ORDER_ID;
-- 3) 标订单 cancelled
UPDATE orders SET status = 'cancelled', updated_at = NOW() WHERE id = $ORDER_ID;
-- 4) 标发票 cancelled（若已生成）
UPDATE invoices SET status = 'cancelled' WHERE order_id = $ORDER_ID;
-- 验证 → COMMIT 或 ROLLBACK
COMMIT;
```

**预防**（不在本 OPS scope，作为未来改进备忘）：把 `Enqueue` 失败时调
`rollbackPayment` 的覆盖范围扩到"已 INSERT vms 但 enqueue 失败"，或在
PayWithBalance 同事务内插一个 `pending_enqueue` 标记，sweeper 兼扫。
