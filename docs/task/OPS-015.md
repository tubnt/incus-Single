## 现象

OPS-008 A1：`/admin/monitoring` 图表 + 表格只显示 `vm-cdc154` 一台，其他 4 台 running VM（MyVM、vm-35762f、vm-73a439、vm-784367）全部缺失。

## 根因（生产 5 节点集群实测）

`incus_admin` 后端 `MetricsHandler.fetchVMs` 调用 `client.RawGet(ctx, "/1.0/metrics")`。Incus 集群该端点的语义是：**只返回当前响应该请求的 cluster member 的本地节点 VM**——不是整个集群的聚合。

实测每节点 `/1.0/metrics?target=NODE` 返回：

| 节点 | 返回的 VM |
|---|---|
| node1 | vm-cdc154 |
| node2 | vm-35762f |
| node3 | MyVM, vm-73a439 |
| node4 | （空——无 VM 在此节点） |
| node5 | vm-784367 |

incus-admin 连接的是 `https://10.0.20.1:8443`（node1），所以原代码只看到 vm-cdc154。

## 修复

### `internal/handler/portal/metrics.go::fetchVMs`

改为 fan-out 跨所有 cluster members：

```go
members, mErr := client.GetClusterMembers(r.Context())
allVMs := make(map[string]*VMMetric)

if mErr != nil || len(members) == 0 {
    // 单节点 / members 失败 → 回退到根路径
    text, err := client.RawGet(r.Context(), "/1.0/metrics")
    ...
} else {
    for _, raw := range members {
        var member struct{ ServerName, Status string `json:"server_name,status"` }
        if err := json.Unmarshal(raw, &member); err != nil || member.ServerName == "" {
            continue
        }
        if member.Status != "" && member.Status != "Online" {
            continue  // skip evacuated / offline 节点，避免拖慢仪表盘
        }
        text, err := client.RawGet(r.Context(), "/1.0/metrics?target="+member.ServerName)
        if err != nil {
            slog.Warn("metrics fetch per-node failed", "node", member.ServerName, ...)
            continue
        }
        for name, metric := range parseMetricsForAllVMs(text) {
            allVMs[name] = metric
        }
    }
}

h.cache.set(clusterName, allVMs)
```

特性：
- 30s 缓存仍然生效（避免 N×fetch 频率过高）
- 单节点集群自动走 fallback（不破坏现有部署）
- 节点离线/演练态自动跳过（不阻塞其他节点）
- 同名 VM 跨项目冲突时后写覆盖前写（与原行为一致）
