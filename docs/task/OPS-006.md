## 描述

VM 行为异常时，admin 能"冻结现场" —— 自动快照 + 停机，随后可进原盘挂载检查或者直接 restore 回滚。不是真正的 rescue-ISO boot（那需要换 bootloader，风险太高），但覆盖 80% 实际 rescue 场景。

## 笔记

- **范围调整**：原 PLAN 的"换 root disk + 原盘挂 data"因需真动 bootloader 暂不做；pivot 为 safe-mode-with-snapshot，覆盖实际 rescue 需求的 80%
- **为什么 name-keyed 路由**：既有 reinstall/migrate/evacuate 都用 name，Rescue 用相同语义方便前端统一集成
- **stop 用 force=true**：rescue 存在的场景就是 VM 已经异常，graceful stop 可能挂住；force 保证快速进 rescue
- **restore 路径**：PUT /1.0/instances/{name} body `{"restore":"<snap-name>"}` 触发 Incus 原生 restore；实验过对 vm-a3e86d 真 restore 回来过（2026-04 早期迁移工作中），行为稳定
- **snapshot 清理**：exit 的 `delete_snapshot` 默认 false —— 让 admin 保留 escape hatch，除非主动要求
- **未来扩展点**：现有 rescue_snapshot_name 字段同样能容纳"rescue image"模式的 sentinel（例如 `rescue-iso:<path>`），如需真 ISO boot 可单独立项增量
