## 现象

OPS-007 收尾时遗留：portal `/vm-detail` 没有 Firewall tab，**用户必须靠管理员去 `/admin/firewall` 才能给自己的 VM 绑组**——这违背 PLAN-021 Phase E 的"用户自助"设计。后端 portal API（`GET/POST/DELETE /portal/services/{id}/firewall`）早已就绪（OPS-007 一并落地），缺的只是前端。

## 修复

### `web/src/features/firewall/api.ts`

新增 4 个 portal hook（admin 已有的 `useFirewallGroupsQuery` 等保留不动）：

- `usePortalFirewallGroupsQuery()` — 只读全部组（用户视角，rules 不暴露给普通用户）
- `usePortalVMFirewallBindingsQuery(vmID)` — 拿到该 VM 当前已绑组（owner-scoped，后端自动 403）
- `usePortalBindVMFirewallMutation(vmID)` — 绑一个 group（cold-modify：running VM 自动 stop→PATCH→start）
- `usePortalUnbindVMFirewallMutation(vmID)` — 解绑

### `web/src/app/routes/vm-detail.tsx`

第三个 tab `firewall` 加入 tab 切换；新组件 `PortalFirewallPanel` 渲染：
- "已绑定的防火墙组" 区块（每行 name / slug / description + 解绑按钮）
- "可绑定的防火墙组" 区块（剩余 group + 绑定按钮）
- cold-modify 提示行（"running VM 会自动 stop→PATCH→start，约 10-15s 不可达"）

destructive 按钮全部按 OPS-009 规范：`aria-label="Unbind firewall group <slug>"` / `data-testid="unbind-firewall-<slug>"` / 解绑前 confirm dialog 显式列出 group 名。

### i18n（zh + en）

新增 `vm.firewall.{tab,bound,available,noBound,noGroupsConfigured,allBound,bind,unbind,bindOk,unbindOk,unbindConfirmTitle,unbindConfirmMessage,coldModifyHint}`。
