# UX-006 字号审计 + UI 简化方案调研

- **status**: investigated（待你决策）
- **priority**: P2
- **owner**: claude
- **createdAt**: 2026-05-03
- **scope**: 用户反馈"字体太大、UI 太复杂、不好用"。系统调研字号使用分布 + 社区最佳实践，产出可量化的简化提案

## 核心结论

**两份报告（代码审计 + 社区对标）独立得出同一结论**：IncusAdmin 当前字号系统每档都比 Linear / Vercel / GitHub / Stripe 对标偏大约**一个台阶**。

| 元素 | 我们当前 | Linear | Vercel | GitHub | Stripe | 建议值 |
|---|---|---|---|---|---|---|
| 内页 PageHeader (H1) | **32px** (text-h1) | 14–20px workspace | 24px (--text-xl) | 20–24px | 24px | **24px** |
| 卡片 / 段落标题 (H3) | **20px** (text-h3) ×91 处 | 14–15px | 16–17px (--text-base/lg) | 16px | 14–16px | **15–16px** |
| 数字展示 (价格/余额) | **24px** (text-h2) | 16px | 18px-bold | 14px | 16–18px | **17–18px** (text-body-emphasis) |
| 表格行高 | varies | dense 32–36px | 32–36px | 36px | 32–36px | **36px** |
| Sidebar 项 | varies | 28–30px / 13–14px | 32–36px | 32–36px | 32–36px | **32px / 13px** |
| Form section 段间距 | gap-8 (32px) | 段内 12 / 段间 24 | 16 / 24–32 | 16 / 24 | 16 / 24 | 段内 12, 段间 24 (gap-3 / gap-6) |

## 关键反模式（Agent A 量化）

1. **`<CardTitle>` 硬编码 `text-h3` + `font-strong`** —— 影响 **91 个卡片标题**，全部 20px / 590 weight，是"看着大"的最大单一来源
2. **`<DialogTitle>` / `<SheetTitle>` 同问题** —— 8 + 2 处对话框/抽屉标题
3. **数字展示 `text-h2` 24px** —— `dashboard 余额`、`launch 月费`、`billing 金额` 三处
4. **`<FormSection>` 标题 `text-h3` 20px** —— `/launch` 4 段都是 20px
5. **`PageShell gap-6` + `LaunchPage gap-8`** —— 顶级 + 段间间距偏大
6. **`CardHeader p-5`** —— 内边距 20px 偏松（dense 应 16px）

## Agent B 调研要点（对应 Source 见任务文件链接列表）

- **Vercel Geist** 内页用 24px，48-64px 仅留给 marketing landing
- **Linear** 产品 UI **不存在 32px+ 标题**；workspace 内最大 20px
- **Stripe Dashboard** "headers 24px / body 14-16px / table 13px"
- **GitHub Primer** PageHeader 标题 20-24px
- **MUI DataGrid** dense 36px / standard 52px，Pencil & Paper 推荐企业表格让用户切 density
- **Atlassian Spacing**：space.150-300 (12-24px) 用于 dense UI，space.400+ (32px+) 仅 layout 级

## 简化提案（请选）

### 方案甲 · 仅字号 token 全局回调一档（**最小成本，立竿见影**）
**改动**：`index.css` `@theme` 一处，几行：

```diff
- --text-h1: 2rem;        /* 32px */
+ --text-h1: 1.5rem;      /* 24px */
- --text-h2: 1.5rem;      /* 24px */
+ --text-h2: 1.25rem;     /* 20px */
- --text-h3: 1.25rem;     /* 20px */
+ --text-h3: 1rem;        /* 16px */
```

附属调整（同 token 文件）：
- `<CardTitle>` 默认 weight 从 `font-strong` 改 `font-emphasis`（对应 590 → 510，与 Linear 对齐）
- 表格 column header 字号 `text-label`(12px) → `text-sm`(14px)，提高扫读性

**收益**：
- 91 个卡片标题、所有 PageHeader、所有 Dialog/Sheet 标题瞬间回到对标水准
- 0 业务代码改动，0 行为变化
- 视觉回归测试：跑 Playwright 在 5 个核心页对比 before/after 截图

**风险**：
- 视觉回归 —— 部分页面可能"过紧"，需要在测试时单独提价（极少数情况下打 className 单点回大）
- 不解决"间距偏松"和"IA 复杂"两个问题

**工作量**：~2 小时，单一 PR

### 方案乙 · 甲 + 间距 + 表格密度（**中等成本，进一步紧凑**）

甲的所有改动 + 以下：

1. `index.css` 加新 spacing scale：默认 `gap-3` (12px) / 卡片间用 `gap-4` (16px) / 大区块用 `gap-6` (24px)；`gap-8` 仅保留 layout 级
2. `<CardHeader>` 默认 `p-5` → `p-4`；`<CardContent>` 列表型卡 `p-3`，hero 类 `p-5`（加 variant prop）
3. `<DataTable>` 默认行高 → 36px：`<TableCell>` `py-2` → `py-1.5` + `text-sm`(14px) 替代 `text-label`(12px)
4. Sidebar 项高度压到 `h-8` (32px)，字号 13–14px
5. Launch 页 `gap-8` → `gap-6`（段间），段内字段 `gap-3`

**收益**：
- 整体页面视觉密度回到 Stripe / Linear 量级
- 表格扫读速度提升（更多行可见 + 字号舒适）
- Sidebar 占屏减小 → 主内容区更宽

**风险**：
- 全局 spacing 改动会动 ~50 处文件，需要逐页 visual test
- 移动端可能需要不同的 spacing scale（mobile 偏松 / desktop 偏紧）

**工作量**：~6 小时，可拆 2 个 PR（token + 业务）

### 方案丙 · 甲 + 乙 + IA 简化（**最大成本，重构信息架构**）

乙的所有改动 + 以下：

1. **顶级 sidebar 收口为 7 项**：`Overview / VMs / Storage / Network / Billing / Tickets / Settings`
   - 现 30+ 项 admin/* 全部归并到 `Settings` 二级 + `Network` 二级 + `Storage` 二级
2. **admin 路由进 Settings 子布局**：`/admin/firewall` → `/settings/firewall` 等。Sidebar 第二级 tabs（GitHub repo / Stripe customer 模式）
3. **资源详情统一 tabs**：vm-detail 已是；推广到 `tickets/:id` `invoices/:id` `clusters/:name`，避免新页跳转
4. **Command Palette 升级为主 navigation**：`Cmd+K` 显示完整导航树 + 键盘 first（Linear 模式）。Sidebar 是次要入口

**收益**：
- "30+ 项 sidebar" 这种压迫感彻底消除
- admin / customer 视角清晰分离（你之前的 UX-002 工作的延续）
- 用户主流路径只走 7 个顶级页 + 资源 detail tabs

**风险**：
- 改 routeTree、改 sidebar-data、改大量 link target；需要后端 metadata 协助（angular menu 配置等）
- 改路由 = 改 URL，所有 deep link 需要 redirect alias 兼容期
- 需要重新打磨 i18n 文案（菜单结构变了）
- **本质是产品决策**，不是纯实现

**工作量**：~3-5 个工作日，多个 PR + 灰度发布

## 我的建议

- 你反馈的核心是"**字大、复杂**" → 方案甲解决"大"，方案丙解决"复杂"
- **务实路径**：先 `proceed 甲`（2 小时），上线观察 1-2 天看反馈
- 如果"大"解决了但还觉得"挤迫" / "复杂"，再 `proceed 乙` 或 `proceed 丙`
- 方案丙是 **IA 决策**，建议你看完甲的实际效果后，针对你最不能忍的那部分再说

不建议直接跳丙，因为 IA 改动是不可逆的认知负担（用户要重新学）。先做最便宜的字号 token 回调一档，看效果再说。
