# OPS-010 修复 Pay-with-Balance 重复支付竞态 + 友好化错误信息

- **status**: completed
- **priority**: P1
- **owner**: claude
- **createdAt**: 2026-04-25 12:00
- **completedAt**: 2026-04-25 12:00
- **relatedBug**: 用户报告 "点击余额支付的时候显示：HTTP 400: order not pending"

## 现象

用户在 `/billing` 点击 "Pay with Balance" 购买云主机，看到红色错误 "HTTP 400: order not pending"，以为没买到。但**实际后端已经创建了 VM**，仅是第二次 /pay 请求碰到了竞态。

## 根因（生产 journalctl 时序，order #18）

```
11:43:40.564  POST /portal/orders         → 201 (order #18 pending)
11:43:40.678  GET  /portal/orders         → 200 (refetch — 关键！)
11:43:40.794  POST /portal/orders/18/pay  → 启动（VM provisioning 14.5s 同步）
11:43:42.768  POST /portal/orders/18/pay  → 400 "order not pending"  ← 第 2 次点击
11:43:55.277  POST /portal/orders/18/pay  → 200 (VM vm-784367 创建完成)
```

链路：
1. `ProductCard.submitOrder` 调用 `useCreateOrderMutation` → 后台返回 order #18 (pending)
2. **`useCreateOrderMutation.onSuccess` 同时 `invalidateQueries(orderKeys.all)`** → 触发 `useMyOrdersQuery` 重 fetch
3. 用户的"我的订单"列表里立刻多出 #18 (pending) + 一个 OrderRow 的"Pay"按钮
4. ProductCard 已经在 14.5s 的同步 /pay 中（VM 创建慢），用户等不到看不到反馈，于是点击 OrderRow 的 Pay 按钮
5. 第 2 次 /pay 请求 → 后端 `PayWithBalance` `SELECT FOR UPDATE` 拿到锁 → status 已被前一个事务改成 paid → 返回 `"order not pending"`
6. OrderRow 的 `payMutation.isError = true`，渲染错误（line 308）但**没被 `o.status === "pending"` 守护**——订单后来变 active 了，错误还在
7. 用户看到红色 "HTTP 400: order not pending"，以为没买到

## 修复（3 处）

### Fix #1（关键）：`features/billing/api.ts` — 移除 useCreateOrderMutation 的 invalidateQueries

```diff
 export function useCreateOrderMutation() {
+  // 不再 invalidate orderKeys：create-then-pay 链路中 pay 同步等待 14s VM provisioning，
+  // 中间 refetch 会让 OrderRow 多出 Pay 按钮，引入并发支付竞态。
+  // pay 完成后由 usePayOrderMutation 自己 invalidate 一次即可。
   return useMutation({
     mutationFn: (params: ...) => http.post<...>("/portal/orders", params),
-    onSuccess: () => queryClient.invalidateQueries({ queryKey: orderKeys.all }),
   });
 }
```

### Fix #2（UX）：`routes/billing.tsx` line 308 — OrderRow 错误信息只在 pending 态显示

```diff
- {payMutation.isError && <span className="...">{(payMutation.error as Error).message}</span>}
+ {payMutation.isError && o.status === "pending" && (
+   <span className="...">{(payMutation.error as Error).message}</span>
+ )}
```

### Fix #3（消息友好）：`internal/repository/order.go::PayWithBalance` — 中文 + 区分子状态

```diff
- if o.Status != model.OrderPending {
-   return fmt.Errorf("order not pending")
- }
+ if o.Status != model.OrderPending {
+   switch o.Status {
+   case model.OrderPaid, model.OrderActive:
+     return fmt.Errorf("订单已支付，请刷新页面查看 VM")
+   case model.OrderCancelled:
+     return fmt.Errorf("订单已取消，无法重复支付")
+   default:
+     return fmt.Errorf("订单状态异常（%s），请刷新页面", o.Status)
+   }
+ }
```

测试 `TestPayWithBalance_OrderNotPending` 只断言 `err != nil`，消息变更不破坏单测。

## 笔记

- `useCreateOrderMutation` 的 invalidate 移除后，"创建订单失败但仍想看到订单状态"这类边缘场景受影响——但这种场景不存在：create 失败时 mutation 抛错，UI 显示错误而不会有 stale order
- 同步 14s 的 VM provisioning 仍是 UX 痛点（用户等不到任何反馈）。**长期建议**（非本任务范围）：把 VM provisioning 异步化，/pay 仅扣款 + 入队，VM 创建走 worker 推 SSE 事件
- Bug #14（formatHttpErrorBody）在前端把 body.error 暴露给 toast；本任务把后端那侧从英文换成中文，链条完整
