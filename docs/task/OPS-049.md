# OPS-049 速率限流换 token bucket + IETF RateLimit headers

- **status**: completed
- **priority**: P1
- **owner**: claude
- **createdAt**: 2026-05-10
- **completedAt**: 2026-05-10
- **scope**: 现 `/api/portal` 30/min 固定窗口共享桶被 SPA 顺带请求烧光导致
  正常用户开第 3 台 VM 被 429 拒绝（21:05:20 实测）。按行业基线
  （DigitalOcean / Linode / Vultr / k8s / GCP）切换 token bucket 并
  补 IETF draft RateLimit headers，让客户端能智能退避。

## 触发现场

- tom@5ok.co 21:04:46 / 21:05:11 两次下单 201 → 21:05:20 第 3 次 429
- 21:05:20~33 持续 13 秒所有 `/api/portal/*` GET/POST 全 429
- 原因不是配额：`max_vms=5` / 当前 2 台活跃，配额完全够；
  是 `defaultLimiter = newRateLimiter(30, time.Minute)` 固定窗口共享桶
  被 SPA 一页加载 8-15 个 API 请求快速耗尽

## 行业基线

| 厂商 | per-user 限流 | 算法 |
|---|---|---|
| DigitalOcean | 5000/h + 250/min burst | sliding window |
| Linode | 1600/min（写）/ 200/min（分页 GET）| token bucket |
| Vultr | 1800/min (30 req/s) | token bucket |
| Hetzner | 3600/h IP | token bucket |
| k8s/GCP/Istio | — | token bucket |

共识：
- token bucket 是云原生默认（k8s/GCP/Vultr/Istio）
- 允许短 burst + 强制长期平均，匹配真实 SPA 流量
- burst ≈ 2-3× sustained rate
- 必须返回 `RateLimit-*` (IETF draft) + `Retry-After`
- 固定窗口有跨界 2x burst 漏洞

## 方案

**算法**：fixed window → `golang.org/x/time/rate` token bucket（Go 官方）

**桶参数**（取 DO / Vultr 中位档，匹配私有云负载）：

| 桶 | sustained | burst | 长期等效 | 对照 |
|---|---|---|---|---|
| `/api/portal/*` default | 5 req/s | 30 | 300/min | DO 250/min + |
| sensitive (stepup/users:batch/shadow) | 1 req/s | 5 | 60/min | 当前 5/min 长期一致但允许 5 burst |

**响应 headers**（IETF draft-ietf-httpapi-ratelimit-headers）：
- `RateLimit-Limit`: e.g. `30;w=6` (burst;window)
- `RateLimit-Remaining`: 剩余 token 数
- `RateLimit-Reset`: 秒数到下一次 token refill
- `Retry-After`: 同上但仅在 429 时返

**保留**：
- per-email key（已登录）/ XFF IP（未登录）双策略
- SSE `/stream` 透传不限
- trusted proxy XFF 解析
- sensitive 路由前缀列表
- 测试不变（互斥性 / 非空 / 前缀匹配）

## 风险

| 风险 | 缓解 |
|---|---|
| 提速后真有恶意刷单 | sensitive 桶 + 仍有 max_vms / max_vcpus 等业务配额兜底 |
| header 名大小写 | IETF 推荐小写无前缀 (`ratelimit-*`)，但很多客户端兼容 `X-RateLimit-*`；先 follow IETF 当前主流 |
| 现有 client 不识别 Retry-After | 标准 HTTP 头，浏览器 / curl / SDK 全支持 |
| go 内存 map 不清理 | x/time/rate 自己管 token；外层 visitor map 需保留 cleanup goroutine |

## 实际改动

- `internal/middleware/ratelimit.go` 整体重写：
  - 自定义固定窗口 `rateLimiter` → `keyedLimiter`（包 `golang.org/x/time/rate.Limiter` per-key）
  - portal 桶：5 rps + burst 30（旧 30/min fixed window → 新 ~300/min long-term）
  - sensitive 桶：1 rps + burst 5（旧 5/min fixed → 新 ~60/min long-term + 5 burst）
  - 加 IETF draft RateLimit-Limit / -Remaining / -Reset 头
  - 429 时加 Retry-After（秒数）
  - cleanup goroutine 改 10 分钟扫一次 / 30 分钟空闲删除
